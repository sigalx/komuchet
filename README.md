# КомУчёт

КомУчёт - система для автоматизации учета коммунальных начислений и оплат в хозяйствах: СНТ, ТСЖ и похожих объединениях.

Текущий рабочий контур автоматизирует прием показаний электросчетчиков, ручной учет оплат, расчет электроэнергии, формирование квитанций, PDF/QR и очередь email-доставки.

## Текущий Этап

Базовая persistence-модель MVP зафиксирована в Symfony/Doctrine сущностях и ручных PostgreSQL-миграциях. Auth-основа реализована: вход по email и паролю, глобальное администрирование через `users.admin_granted_at/admin_revoked_at`, роли хозяйства через `workspace_user_role_assignments`, проверка `approved`/`blocked`/`deleted` состояния пользователя.

Админка `/admin` на Bootstrap работает как рабочий стол оператора: показывает открытые проблемы расчетов, дату формирования квитанций, должников, последние оплаты и последние показания. Рабочие разделы - пользователи `/admin/users` со связью с абонентами, выдачей доступа к порталу и ролями хозяйства, участки `/admin/accounts` с вычисляемым балансом, экраном "Баланс и операции" и фиксацией snapshot-квитанции, баланс участков `/admin/account-balances`, абоненты `/admin/subscribers`, группы участков `/admin/account-groups`, электросчетчики `/admin/electricity-meters`, показания `/admin/electricity-meter-readings`, оплаты `/admin/payments`, платежные реквизиты `/admin/payment-requisite-profiles`, тарифные профили `/admin/electricity-tariff-profiles`, тарифные зоны `/admin/electricity-tariff-zones`, диапазоны потребления `/admin/electricity-consumption-bands`, правила диапазонов `/admin/electricity-consumption-band-rules`, тарифные периоды и ставки, расчеты `/admin/billing-runs`, сохраненные квитанции `/admin/account-statements`, доставки `/admin/account-statement-deliveries`, аудит `/admin/audit-logs`, назначение тарифных профилей участкам и двустороннее управление доступом абонентов к участкам.

Личный кабинет `/portal` реализован: абонент видит доступные участки, баланс, истории показаний/начислений/оплат, сохраненные системные квитанции и может передать новые показания активного электросчетчика.

Проект по-прежнему ведется без доступа к реальным Excel-файлам, почтовым ящикам и бухгалтерским сводкам. Обезличенный PDF в `docs/samples/` используется только как пример текущего результата ручного процесса: акт сверки и квитанция по одному участку.

Dev-окружение запускается через Docker Compose: `php-web`, `php-cli`, `composer`, nginx, PostgreSQL 18 и одноразовый Playwright `e2e` runner. Есть изолированный fresh install smoke, который поднимает проект на пустых volumes и прогоняет миграции, PHPUnit и Playwright.

## Цели

- Дать абоненту личный кабинет для передачи показаний и просмотра участка.
- Показывать историю показаний, начислений, оплат и вычисляемый баланс.
- Рассчитывать электроэнергию с учетом тарифов, тарифных зон и социальных норм.
- Дать оператору и администратору ручные инструменты обслуживания абонентов, участков, счетчиков, оплат, реквизитов и расчетов.
- Формировать сохраненные системные квитанции с PDF, QR-кодом оплаты и email-доставкой.
- Сохранять аудит финансово значимых и access-действий.
- Подготовить проект к pilot/prod-запуску.

## Ключевые Решения

- Начисления относятся к `Account`, то есть к участку или лицевому счету, а не напрямую к человеку.
- Один `Subscriber` может иметь доступ к нескольким `Account`.
- Все входы в систему идут через единую сущность `User`; глобальные администраторы отмечаются admin lifecycle-полями, роли внутри хозяйства хранятся как workspace-scoped назначения `admin`/`operator`.
- Primary keys по умолчанию - UUID, предпочтительно UUIDv7 в PostgreSQL 18.
- Абсолютные моменты времени хранятся в `timestamp with time zone` / `timestamptz`.
- Показания электросчетчика хранятся как события, а не как строго месячные записи.
- Баланс должен быть вычисляемым значением на основе начислений и оплат.
- Тарифы и социальные нормы нужно хранить исторически, потому что их могут исправлять задним числом.
- Сумма в квитанции является текущим долгом или остатком к оплате на дату формирования, а не только начислением за последний месяц.
- Первый интерфейс делается как Symfony-приложение без отдельного frontend.
- MVP использует email + пароль. Телефон и внешние провайдеры аутентификации закладываются как будущие расширения.
- Реальные персональные данные и платежные реквизиты не должны попадать в документацию без необходимости.

## Технологический Стек

- Backend: PHP + Symfony.
- Frontend: серверные страницы Symfony/Twig, без отдельного SPA.
- Database: PostgreSQL.
- Версии: актуальные стабильные версии на момент старта проекта.

## Локальная Разработка

Минимальный dev-контур запускается через Makefile:

```bash
make init
make up
make migrate
make bootstrap-workspace
make create-admin ARGS="admin@example.test"
```

По умолчанию HTTP доступен на `127.0.0.1:8080`, PostgreSQL проброшен на `127.0.0.1:5433`. Локальные значения можно переопределить в `.env.dev.local`.

## Быстрый UI Smoke

Для проверки dev-окружения с демо-данными и браузерным smoke-тестом:

```bash
make up
make migrate
make e2e-smoke-full
```

Команда `make e2e-smoke-full` пересоздает демо-хозяйство `demo-smoke`, выдает smoke-пользователю доступ в админку и личный кабинет, затем запускает Playwright-проверку основных UI-сценариев.

## Fresh Install Smoke

Проверка старта с пустых volumes выполняется в отдельном Compose-проекте и не трогает основной dev-контур:

```bash
make fresh-install-smoke
```

По умолчанию проверочный контур доступен на `127.0.0.1:18080`, PostgreSQL проброшен на `127.0.0.1:15433`. Команда применяет миграции, создает хозяйство и администратора, проверяет `/healthz`, `/login`, PHPUnit и Playwright smoke. Очистка отдельного smoke-контура:

```bash
make fresh-install-smoke-down
```

## Тесты

Новый функционал должен сопровождаться тестами. Тестовая БД создается отдельно от dev-БД через стандартный Doctrine suffix `_test`.

```bash
make test
```

Команда создает test-БД при необходимости, применяет миграции и запускает PHPUnit внутри PHP-контейнера.
