# AGENTS.md

Инструкции для Codex-агентов и других автоматизированных помощников, работающих в этом проекте.

## Рабочий Контекст

Проект находится на стадии подготовки вертикального MVP к pilot/prod-запуску. Базовая доменная persistence-модель перенесена в Doctrine-сущности и ручные PostgreSQL-миграции; административный интерфейс, личный кабинет абонента, ручные оплаты, расчеты электроэнергии, snapshot-квитанции, PDF/QR, очередь email-доставки, demo seed, PHPUnit и Playwright smoke уже реализованы.

Язык документации и пользовательского общения - русский.

## Важные Ограничения

- Не переносить персональные данные из `report.pdf` в документацию без прямой необходимости.
- Не считать `report.pdf` исчерпывающей спецификацией. Это пример результата текущего ручного процесса.
- Не предполагать наличие реальных Excel-таблиц, почтового ящика или банковских сводок.
- Оригинальные документы с персональными данными не хранить в репозитории. Для примеров использовать только обезличенные или redacted-копии.
- Если правило расчета не подтверждено, не выдавать его за факт; сначала уточнить с пользователем.
- Баланс считать производным от начислений и оплат, пока не принято отдельное решение о кэшировании.
- Считать `Account` объектом начислений хозяйства; для СНТ это участок. Номер участка хранить текстом.
- Для аутентификации использовать единую сущность `User`. Не проектировать отдельную таблицу логинов для администраторов.
- Правила PostgreSQL-схемы описаны в [docs/database-design.md](docs/database-design.md): UUID PK, `timestamptz` для абсолютных моментов времени, audit-поля и триггеры.
- Согласованная схема MVP описана в [docs/database-schema.md](docs/database-schema.md). При проектировании сущностей сверяться с ней и с уже созданными миграциями.
- Symfony-артефакты генерировать штатными Symfony/Maker-командами, затем дорабатывать вручную там, где нужны PostgreSQL-специфичные ограничения.
- Ручные миграции являются источником истины для PostgreSQL enum, partial indexes, composite FK, exclusion constraints и triggers.
- Dev-СУБД запускается через Docker Compose, а не устанавливается на хост. Базовые команды находятся в `Makefile`: `make init`, `make up`, `make migrate`, `make console ARGS="..."`.
- Dev-контур разделен на `php-web`, `php-cli`, `composer` и `e2e`. Symfony console, миграции, генераторы и PHPUnit запускать через `php-cli`; Composer - через отдельный `composer` tool-сервис; Playwright smoke - через Docker Compose service `e2e`.
- Production compose тоже содержит `php-cli` на том же immutable image, что и `php-web`. Production `bin/console` команды, миграции, bootstrap и scheduled jobs запускать через `php-cli`, а не через web/FPM container.
- Fresh install проверяется командой `make fresh-install-smoke`: она использует отдельный Compose project `snt_fresh_smoke`, отдельные порты и отдельные volumes.
- Не считать миграции примененными, пока это явно не проверено на PostgreSQL в Docker.

## Доменная Терминология

Использовать текущие имена сущностей:

- `User` - учетная запись для входа в систему.
- `Subscriber` - абонент, субъект доступа в систему.
- `Workspace` - хозяйство: изолированный контур бизнес-данных.
- `WorkspaceUserRoleAssignment` - административная роль пользователя внутри хозяйства: `admin` или `operator`.
- `Account` - объект начислений; в текущем СНТ это участок.
- `SubscriberAccountAccess` - связь абонента с доступными ему участками.
- `ElectricityMeter` - электросчетчик.
- `ElectricityMeterReading` - событие передачи показаний электросчетчика.
- `ElectricityTariffPeriod` - период действия тарифов.
- `ElectricityConsumptionBandRule` - правило распределения потребления по диапазонам; социальная норма является частным случаем такого правила.
- `Accrual` - обобщенное начисление по участку.
- `ElectricityAccrualContext` - примененный расчетный контекст электроэнергии для начисления.
- `ElectricityAccrualRegister` - показания, использованные в расчете по регистру счетчика.
- `ElectricityAccrualLine` - финансовая строка начисления электроэнергии.
- `Payment` - поступление денег на баланс участка.
- `PaymentRequisiteProfile` - профиль платежных реквизитов хозяйства.
- `BillingRun` - расчетный запуск за период.
- `AccountStatementSnapshot` - сохраненная системная квитанция со snapshot-суммами, строками, реквизитами, PDF/QR.
- `AccountStatementDelivery` - бизнес-доставка сохраненной квитанции получателю.
- `AccountStatementDeliveryAttempt` - транспортная попытка доставки.
- `AuditLog` - журнал действий и исправлений.

Аутентификация и выдача доступа описаны в [docs/auth-and-access.md](docs/auth-and-access.md).

## Правила Работы

- При изменении документации поддерживать ссылки в `docs/README.md`.
- Новые спорные решения сначала обсуждать с пользователем.
- Для проектных правок предпочитать небольшие, связные изменения.
- Новый функционал сопровождать тестами. Если тесты временно невозможны, явно фиксировать причину и остаточный риск.
- В PHP-сравнениях не использовать Yoda conditions: переменная или вычисляемое выражение должно быть слева, expected/literal/enum value - справа, например `$value === false`, а не `false === $value`.
- После успешного submit CRUD-формы создания или редактирования перенаправлять пользователя на view измененной записи, а не на index. После удаления допустим redirect на index, потому что view удаленной записи не должен открываться.
- Не удалять исходные артефакты из корня проекта без прямой просьбы пользователя.

## Технический Стек

Базовый стек проекта:

- PHP.
- Symfony.
- PostgreSQL.

Использовать актуальные стабильные версии, зафиксированные в [docs/technical-stack.md](docs/technical-stack.md). Перед созданием или обновлением зависимостей проверять, не устарели ли точные patch-версии.

Отдельный frontend не планируется. По умолчанию проектировать интерфейс как серверное Symfony-приложение с Twig.
