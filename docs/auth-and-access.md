# Аутентификация И Доступы

Документ фиксирует решения по пользователям, ролям и выдаче доступа к участкам.

Статус: auth-блок согласован для MVP.

## Основное Решение

Для входа в систему используется единая сущность `User`.

Не нужно делать отдельные таблицы логинов для абонентов и администраторов. Администратор - это такой же `User`, но с административной ролью.

Причины:

- Symfony Security естественно работает с одной сущностью пользователя;
- один человек может одновременно быть абонентом и администратором;
- проще добавлять TOTP, блокировки, аудит входов и внешние провайдеры;
- не нужно дублировать email, телефон, парольные политики и восстановление доступа.

## User

`User` отвечает за жизненный цикл учетной записи. Все UUID-колонки в БД называются явно: primary key `uuid`, внешние ключи `<entity>_uuid`.

`users` не хранит пароль, роли и динамическую историю входов. Эти данные вынесены в отдельные таблицы.

- `created_at`;
- `updated_at`;
- `approved_at`;
- `approved_by`;
- `blocked_at`;
- `blocked_reason`;
- `blocked_by`;
- `deleted_at`;
- `deleted_by`;
- `created_by`;
- `updated_by`.

Состояние пользователя выводится из дат:

- `pending` - `approved_at is null`;
- `active` - `approved_at is not null`, `blocked_at is null`, `deleted_at is null`;
- `blocked` - `blocked_at is not null`, `deleted_at is null`;
- `deleted` - `deleted_at is not null`.

Удаленный пользователь не может проходить login.

Таблица `users` согласована без отдельного `status`: состояние выводится из дат. `updated_at`/`updated_by` остаются, потому что жизненный цикл учетной записи изменяется после создания.

## Email Identities

Email хранится отдельно от `users` в `user_email_identities`.

Поля:

- `user_uuid`;
- `email` - исходное значение email;
- `email_normalized` - нормализованное значение для поиска и unique index;
- `verified_at` - момент подтверждения email.
- `created_at`;
- `deleted_at`.

Primary key: `(user_uuid, email_normalized)`.

Отвязка email выполняется soft-delete через `deleted_at`, без физического удаления записи. Активная запись - `deleted_at is null`.

Unique index строится только по активным email:

```sql
unique (email_normalized) where deleted_at is null
```

Это позволяет отвязать email от одного пользователя и привязать тот же email другому пользователю без потери истории.

`user_email_identities` согласована: один пользователь может иметь несколько активных email, но один активный email не может быть привязан к нескольким пользователям.

Телефон можно хранить как контакт абонента и позже подключить отдельной таблицей `user_phone_identities`.

VK, Яндекс и Госуслуги добавляются отдельной таблицей внешних идентичностей после MVP.

## Password Credentials

Локальный пароль хранится отдельно в `user_password_credentials`:

- `user_uuid` - primary key и ссылка на `users.uuid`;
- `password_hash` - текущий hash локального пароля;
- `changed_at` - когда текущий пароль был установлен;
- `expires_at` - срок действия текущего password credential.

`expires_at is null` означает, что текущий пароль не протухает по времени.

`created_at` в `user_password_credentials` не нужен: он дублировал бы `changed_at`.

`changed_by` в `user_password_credentials` не хранится: текущая таблица не является audit-журналом, а инициатор смены пароля хранится в `user_password_history`.

История паролей хранится отдельно в `user_password_history`. При каждой установке нового локального пароля приложение:

- обновляет текущую запись в `user_password_credentials`;
- добавляет append-only запись в `user_password_history`.

`user_password_history` хранит:

- `user_uuid` - чей пароль был изменен;
- `password_hash` - password hash, установленный в момент изменения;
- `changed_at` - время изменения;
- `changed_by` - кто инициировал изменение, если известен.

Отдельный `uuid` истории не нужен, пока нет сценария адресовать одну запись истории напрямую. История нужна для аудита и будущего запрета повторного использования старых паролей.

`user_password_history` согласована как append-only таблица без surrogate `uuid`, без `created_at`/`updated_at`/`deleted_at` и без `expires_at`.

## Subscriber

`Subscriber` - доменная сущность абонента хозяйства.

Абонент может быть связан с `User`, если у него есть личный кабинет. При этом абонент может существовать и без активной учетной записи: это нужно для жителей без интернета, за которых показания вносит администратор.

ФИО хранится раздельно: `last_name`, `first_name`, `second_name`. `second_name` nullable, `full_name` не хранится.

Связь с участками хранится через `SubscriberAccountAccess`.

## Права И Роли

В модели доступа есть три независимых слоя.

Глобальный администратор платформы хранится в `users`: пользователь является админом, если `admin_granted_at is not null and admin_revoked_at is null`. Отдельный boolean `admin` не нужен. На первом этапе выдача и отзыв этого признака идут через CLI или ручное серверное действие, без UI.

Административные права внутри хозяйства хранятся в `workspace_user_role_assignments`:

- `admin` - администратор конкретного хозяйства;
- `operator` - оператор конкретного хозяйства.

Абонентский доступ не хранится как роль. Он выводится из связи `subscribers.user_uuid` внутри хозяйства. Если пользователь привязан к активному `Subscriber`, он видит абонентский портал этого хозяйства и участки через `SubscriberAccountAccess`.

Один человек может одновременно быть абонентом и оператором/администратором того же хозяйства. Это нормальный сценарий: оператор сам может жить в этом хозяйстве и пользоваться абонентским порталом.

Права выдачи:

- глобальный администратор управляет всеми хозяйствами;
- администратор хозяйства может выдавать роли `admin` и `operator` внутри своего хозяйства;
- оператор хозяйства работает с данными хозяйства и подключает новых абонентов, но не выдает административные роли;
- абонент ничего не выдает, только пользуется абонентским порталом.

Identity-управление учетными записями в разделе `/admin/users` требует `WORKSPACE_ADMIN`: создание пользователя, изменение email identities, одобрение, блокировка, удаление, установка пароля, прямая связь `User` с `Subscriber` из карточки пользователя и управление ролями хозяйства. Оператор хозяйства может смотреть пользователей и подключать абонентский портал из карточки `Subscriber`, но не управляет глобальной учетной записью напрямую.

Активная роль хозяйства - запись `workspace_user_role_assignments`, у которой `revoked_at is null`.

`workspace_user_role_assignments` согласована без технических `created_at`/`updated_at`: для этой таблицы достаточно `granted_at`/`granted_by` и `revoked_at`/`revoked_by`.

Админка `/admin` защищается правом `WORKSPACE_ACCESS`: глобальный администратор, администратор хозяйства и оператор хозяйства могут входить в административный интерфейс текущего хозяйства.

Текущее хозяйство хранится в Symfony session. Если выбранное хозяйство отсутствует или больше не доступно пользователю, `WorkspaceContext` выбирает первое доступное хозяйство. Глобальный администратор видит все хозяйства; администраторы и операторы хозяйств видят только хозяйства, где у них есть активная роль.

Действия управления ролями хозяйства и identity-действия пользователей защищаются правом `WORKSPACE_ADMIN`: глобальный администратор и администратор текущего хозяйства могут выдавать и отзывать роли `admin`/`operator`, менять учетные записи и напрямую связывать их с абонентами; оператор хозяйства не может.

Просмотр audit log также защищается правом `WORKSPACE_ADMIN`, но выборка зависит от уровня пользователя:

- глобальный администратор видит все `audit_logs`, включая глобальные события без `workspace_uuid`;
- администратор хозяйства видит только `audit_logs` текущего хозяйства;
- оператор хозяйства и абонент audit log не видят.

## Вход В MVP

MVP использует email и пароль.

Телефон хранится как контакт и будущий способ входа, но вход по SMS или одноразовому коду не входит в первый этап.

Свободная самостоятельная регистрация не входит в ближайший этап. Доступ выдается через бизнес-действие подключения email к хозяйству.

MVP использует стандартные Symfony sessions и не вводит доменную таблицу сессий или отдельный login-event audit. Если позже понадобятся управляемые сессии или security audit попыток входа, их нужно проектировать отдельными таблицами.

Реализация входа:

- login/logout сгенерированы штатным `make:security:form-login`;
- форма входа принимает email в поле `_email`;
- `App\Security\UserEmailProvider` ищет только активную и подтвержденную запись `user_email_identities`;
- `User` перезагружается по `uuid` при восстановлении пользователя из session;
- глобальный admin-признак берется из `users.admin_granted_at`/`admin_revoked_at`;
- роли хозяйства берутся из `workspace_user_role_assignments`, где `revoked_at is null`;
- абонентский доступ к хозяйству берется из активных `subscribers`, где `user_uuid` равен текущему пользователю;
- `App\Security\UserAccountChecker` запрещает вход для `pending`, `blocked` и `deleted` пользователей.

Первый администратор создается CLI-командой:

```bash
make create-admin ARGS="admin@example.test"
```

Команда интерактивно запросит пароль, создаст approved `User`, verified `user_email_identities`, текущий `user_password_credentials`, append-only запись `user_password_history` и активный глобальный admin-признак.

По умолчанию команда завершится ошибкой, если активный email уже привязан к пользователю. Для idempotent-сценариев подготовки dev/smoke-данных можно использовать `--if-exists=skip`:

```bash
make create-admin ARGS="admin@example.test --password=temporary-pass-123 --if-exists=skip"
```

## Профиль Пользователя

MVP-профиль доступен на `/profile` любому аутентифицированному пользователю.

Профиль в MVP показывает:

- состояние учетной записи;
- UUID пользователя;
- активные email identities в режиме read-only;
- дату установки текущего локального пароля;
- доступы к хозяйствам: глобальный администратор, роли хозяйства и абонентская привязка через `Subscriber`;
- ссылку на добровольную смену пароля `/password/change`;
- ссылку в личный кабинет, если есть абонентский доступ;
- ссылку в админку, если есть административный доступ.

Самостоятельное управление email identities в MVP не реализуется. Причина: смена или добавление email требует отдельного verification-flow с токенами, сроками действия, повторной отправкой письма и обработкой конфликтов активных email. До появления этого flow email меняет администратор через админку.

Смена пароля доступна всегда, а не только при протухшем временном пароле. При добровольной смене пользователь должен указать текущий пароль, новый пароль и повтор нового пароля. После добровольной смены пользователь возвращается в профиль. Forced-flow для протухшего пароля после успешной смены возвращает на общий dashboard, чтобы дальше сработал обычный выбор портала или админки.

## Выдача Доступа К Хозяйству

Текущий сценарий - управляемая выдача доступа оператором или администратором. У конторы уже есть список email владельцев участков, поэтому пользователю не нужно самостоятельно создавать заявку.

### Подключение Администратора Или Оператора

Администратор хозяйства выполняет бизнес-действие "выдать роль в текущем хозяйстве по email".

Если `User` с таким активным email уже существует, система выдает ему роль `admin` или `operator` в текущем хозяйстве. Пароль и существующие учетные данные не меняются.

Если `User` не существует, система создает глобального пользователя, verified email identity и временный пароль, после чего выдает роль в текущем хозяйстве.

Если существующий `User` еще не одобрен или не имеет локального пароля, система одобряет пользователя и выдает временный пароль. Временный пароль получает `expires_at = '1970-01-01 00:00:00+00'`.

Оператор хозяйства не может выдавать административные роли хозяйства.

### Подключение Абонента

Оператор выполняет бизнес-действие "подключить email к абоненту в хозяйстве".

Если `User` с таким активным email уже существует, система привязывает его к `Subscriber` текущего хозяйства. Пароль и существующие учетные данные не меняются.

Если `User` не существует, система создает глобального пользователя, email identity, временный пароль и привязывает пользователя к `Subscriber` текущего хозяйства.

На выходе новый пользователь получает email и временный пароль. Временный пароль создается заведомо протухшим через `user_password_credentials.expires_at = '1970-01-01 00:00:00+00'`, чтобы после первого входа перевести пользователя в будущий flow смены пароля.

Flow смены временного пароля реализован как `/password/change`. Пользователь с протухшим локальным паролем может успешно пройти login, но до смены пароля request-listener перенаправляет его с остальных защищенных страниц на форму смены пароля. После успешной смены новый `user_password_credentials.expires_at` становится `NULL`, а факт смены пишется в `audit_logs` без сохранения password hash.

### Уведомления

Новому пользователю нужно безопасно передать учетные данные. Если безопасный транспорт еще не настроен, MVP может показать временные креды оператору один раз.

Существующему пользователю при выдаче доступа к новому хозяйству отправляется уведомление: ему выдан доступ к такому-то хозяйству.

### Будущая Самозаявка

Свободную регистрацию можно добавить позже отдельным сценарием: человек сам вводит данные, получает статус ожидания, а администратор после проверки документов одобряет учетную запись и привязывает участки. В MVP этот сценарий не реализуется.

## Доступ К Участкам

`SubscriberAccountAccess` определяет, какие участки доступны абоненту.

Один абонент может иметь доступ к нескольким участкам. Модель также допускает несколько абонентов с доступом к одному участку.

В MVP любой активный доступ позволяет видеть участок и передавать показания. Поле `access_role` хранится для будущего разделения прав: владелец, представитель, наблюдатель.

`access_role` хранится как PostgreSQL enum `subscriber_account_access_role`: `owner`, `representative`, `viewer`.

`subscriber_account_accesses` согласована без surrogate `uuid` и без технических `created_at`/`updated_at`: primary key `(workspace_uuid, subscriber_uuid, account_uuid, granted_at)`, историю описывают `granted_*` и `revoked_*`.

## Административная Двухфакторная Аутентификация

Для администраторов нужно предусмотреть TOTP-двухфакторную аутентификацию, совместимую с приложениями вроде Google Authenticator.

TOTP не входит в MVP-схему. Это вероятное развитие функционала аутентификации. Если TOTP будет добавляться позже, credential нужно хранить отдельно от `users`, например в `user_totp_credentials`.

## Внешние Провайдеры

В будущем нужно иметь возможность подключить внешние сервисы аутентификации:

- VK;
- Яндекс;
- Госуслуги;
- другие провайдеры.

Для этого внешние идентичности должны храниться отдельно от `User`.

Пример полей:

- `user_uuid`;
- `provider`;
- `external_id`;
- `email_from_provider`;
- `created_at`;
- `disabled_at`.

Внешние провайдеры не входят в MVP.

## Отложенное Управление Identity И Sessions

Будущий блок полноценного управления identity должен включать:

- добавление нового email пользователем через подтверждающее письмо;
- смену primary email после подтверждения нового адреса;
- отвязку старого email с защитой от потери последнего активного verified email;
- хранение внешних identity для VK, Яндекса, Госуслуг и других провайдеров;
- подключение TOTP для администраторов и, возможно, для обычных пользователей;
- резервные коды восстановления TOTP;
- аудит всех изменений identity без записи секретов;
- UI просмотра активных серверных сессий;
- отзыв отдельных сессий пользователем или администратором;
- глобальный logout со всех устройств;
- security audit входов, неуспешных попыток, смены пароля и сброса 2FA.

Если появится управляемое хранение сессий, отдельная таблица login events может не понадобиться: активные и завершенные сессии покроют большую часть сценариев. Security audit попыток входа все равно стоит проектировать отдельно, если понадобится анализ неуспешных попыток или подозрительной активности.
