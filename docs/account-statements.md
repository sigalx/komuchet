# Квитанции И Statements

Документ фиксирует текущий контур квитанций: динамический экран сверки по участку, persistent snapshot, печатную форму, PDF и email-доставку.

Статус: реализован read-only экран, сохранение snapshot из админки, snapshot платежных реквизитов, QR-код оплаты, печатная HTML-форма snapshot, PDF-генерация, постановка email-доставки в очередь и SMTP-обработчик queued-попыток.

## Термины

В пользовательском интерфейсе слово "квитанция" используем только для сохраненного snapshot-документа, у которого есть номер, PDF/QR и lifecycle.

Динамический экран сверки в личном кабинете называется "Баланс и операции". В коде для него все еще используется `statement`, потому что это read-model по начислениям, оплатам и балансу.

Сохраненный snapshot является зафиксированной квитанцией с номером, totals, PDF/QR и историей постановки доставки в очередь.

## Текущий Экран

Маршруты:

- `GET /admin/accounts/{uuid}/statement` - просмотр оператором из карточки участка;
- `GET /portal/accounts/{uuid}/balance` - просмотр абонентом текущего экрана "Баланс и операции" по доступному участку.
- `POST /admin/accounts/{uuid}/statements` - зафиксировать текущий statement как snapshot;
- `GET /admin/accounts/{uuid}/statements/{statementUuid}` - просмотр сохраненного snapshot.
- `GET /admin/accounts/{uuid}/statements/{statementUuid}/print` - печатная HTML-форма сохраненного snapshot.
- `GET /admin/accounts/{uuid}/statements/{statementUuid}/pdf` - PDF сохраненного snapshot.
- `POST /admin/accounts/{uuid}/statements/{statementUuid}/deliveries/enqueue` - поставить доставку snapshot в очередь по активным абонентам участка.
- `GET /admin/account-statements` - общий список сохраненных квитанций текущего хозяйства.
- `GET /portal/accounts/{uuid}/statements/{statementUuid}` - просмотр абонентом сохраненного active snapshot по доступному участку.
- `GET /portal/accounts/{uuid}/statements/{statementUuid}/pdf` - PDF сохраненного active snapshot для абонента.

Экран показывает:

- хозяйство;
- номер участка;
- дату формирования экрана;
- сумму active posted начислений;
- сумму active оплат;
- текущий баланс;
- сумму к оплате;
- список active posted начислений;
- список active оплат;
- детализацию расчета электроэнергии, если у начисления есть `electricity_accrual_registers` и `electricity_accrual_lines`.

В текущем read-model не показываются:

- draft-начисления;
- cancelled начисления;
- superseded начисления;
- cancelled оплаты;
- superseded оплаты.

Это совпадает с правилами вычисляемого баланса.

## Read-Model

Текущий экран строится сервисом `AccountStatementProvider`.

Источники данных:

- `AccountBalanceCalculator` - итоговые суммы начислений, оплат и баланса;
- `AccrualRepository::findActivePostedByAccount()` - начисления, участвующие в statement;
- `PaymentRepository::findActiveByAccount()` - оплаты, участвующие в statement;
- `ElectricityAccrualRegisterRepository::findByAccrual()` - расчетные показания по электроэнергии;
- `ElectricityAccrualLineRepository::findByAccrual()` - строки расчета по тарифным зонам и диапазонам.

Сумма к оплате:

```text
amount_to_pay = max(active_posted_accruals - active_payments, 0)
```

Баланс при этом считается как `active_payments - active_posted_accruals`: положительное значение означает переплату, отрицательное - задолженность. Если участок имеет переплату, `amount_to_pay` равен `0.00`, а переплата показывается в балансе.

## Snapshot

Сохраненный statement является финансовым snapshot: после формирования он не должен менять суммы при последующих исправлениях начислений, оплат или тарифов.

Таблицы:

- `account_statements` - заголовок snapshot, номер, участок, дата, totals и lifecycle формирования/отмены;
- `account_statement_accruals` - строки начислений, попавшие в snapshot, со ссылкой на исходное начисление и копией ключевых полей;
- `account_statement_payments` - строки оплат, попавшие в snapshot, со ссылкой на исходную оплату и копией ключевых полей.
- `account_statement_electricity_registers` - snapshot показаний по регистрам счетчика;
- `account_statement_electricity_lines` - snapshot строк расчета электроэнергии.
- `payment_requisite_profiles` и `payment_requisite_assignments` - справочник реквизитов, default-назначение реквизитов для хозяйства и назначения по типам начислений.
- `account_statement_deliveries` - delivery job по конкретной сохраненной квитанции и получателю;
- `account_statement_delivery_attempts` - попытки доставки, включая queued/sending/sent/failed lifecycle.

В snapshot попадают те же данные, что и в динамический statement на момент формирования:

- active posted начисления;
- active оплаты;
- сохраненные totals;
- `amount_to_pay = max(-balance, 0)`;
- `overpayment_amount = max(balance, 0)`.
- платежные реквизиты, действующие на дату квитанции, если для хозяйства назначен подходящий профиль реквизитов.

Отмена или замена исходного начисления/платежа после формирования не меняет сохраненные totals и строки snapshot. Новый корректный документ должен формироваться отдельным snapshot.

В snapshot сохраняются шапка, totals, базовые строки начислений/оплат и детализация электроэнергии:

- тарифная зона;
- модель и серийный номер счетчика, если заполнены;
- предыдущее и текущее показание с датами;
- диапазон потребления;
- расход, ставка и сумма строки.

Названия тарифных зон и диапазонов хранятся копией, чтобы уже сформированная квитанция не менялась при последующем переименовании справочников.

Платежные реквизиты тоже копируются в `account_statements`: получатель, ИНН/КПП, банк, БИК, корреспондентский счет, расчетный счет и назначение платежа. Ссылка на `payment_requisite_profile_uuid` сохраняется для аудита источника, но печатная форма читает именно snapshot-поля.

Выбор реквизитов выполняется через `payment_requisite_assignments`: если все строки начислений в квитанции относятся к одному `AccrualType`, система сначала ищет назначение для этого типа. Если такого назначения нет или квитанция смешанная по типам начислений, используется default-назначение с `accrual_type = null`.

## QR-Код Оплаты

QR-код строится из сохраненных snapshot-реквизитов квитанции, а не из текущего справочника реквизитов.

Формат payload:

```text
ST00012|Name=...|PersonalAcc=...|BankName=...|BIC=...|CorrespAcc=...|PayeeINN=...|KPP=...|Purpose=...|Sum=...
```

`Sum` хранится в копейках. QR-код не формируется, если нет обязательных реквизитов или `amount_to_pay = 0.00`.

Генерация изображения выполняется библиотекой `endroid/qr-code` в PNG data URI. Для этого PHP-окружение должно иметь расширение `ext-gd`. PDF использует тот же `AccountStatementPaymentQrCodeGenerator`, чтобы печатная HTML-форма и PDF кодировали одинаковые данные.

## Печатная HTML-Форма

Печатная форма строится только от сохраненного snapshot. Она не вызывает `AccountStatementProvider` и не читает текущие active начисления/оплаты, поэтому повторное открытие старой печатной формы показывает те же суммы и строки, что были сохранены в момент формирования snapshot.

Маршрут печатной формы не использует административный shell: нет бокового меню, верхней панели и flash-сообщений. На экране есть только служебная панель с кнопками "Назад" и "Печать"; при печати она скрывается CSS-правилом `.no-print`.

## PDF

PDF строится из тех же snapshot-данных, что и HTML-форма: заголовок, totals, строки начислений/оплат, детализация электроэнергии, платежные реквизиты и QR-код оплаты.

Генерация выполняется через `dompdf/dompdf`. Для кириллицы используется встроенный в Dompdf шрифт DejaVu Sans. PDF-шаблон отдельный (`account_statement/pdf.html.twig`) и использует простую табличную верстку, потому что PDF-рендеры хуже поддерживают современный CSS, чем браузеры.

Ответ маршрута возвращается как `application/pdf` с inline `Content-Disposition`.

## Доставка

Администратор на карточке сохраненной квитанции нажимает "Поставить в очередь отправки", после чего система:

- находит активных абонентов участка;
- выбирает email получателя: сначала `subscribers.contact_email`, затем активный verified email связанного `User`;
- пропускает абонентов без email;
- не создает дубль, если для этой квитанции, канала и email уже есть активная отправка;
- создает `account_statement_deliveries` и первую `account_statement_delivery_attempts` со статусом "В очереди".

Статус доставки вычисляется из lifecycle-полей:

- delivery cancelled, если заполнен `cancelled_at`;
- attempt queued, если заполнен `queued_at`, но нет `started_at`, `succeeded_at`, `failed_at`;
- attempt sending, если заполнен `started_at`, но нет terminal timestamp;
- attempt sent, если заполнен `succeeded_at`;
- attempt failed, если заполнен `failed_at`.

SMTP-обработчик запускается командой:

```bash
make send-statement-deliveries ARGS="--limit=50"
```

Команда берет queued attempts, формирует PDF через существующий renderer, отправляет письмо через Symfony Mailer и заполняет `started_at`, `succeeded_at`/`failed_at`, `provider_message_id` или `failure_reason`.

Отмена квитанции выполняется как бизнес-операция над `account_statements`: заполняются `cancelled_at`, `cancelled_by`, `cancellation_reason`. Вместе с квитанцией система отменяет все ее активные `account_statement_deliveries`. Таблица `account_statement_delivery_attempts` остается транспортным журналом и при отмене не меняется, даже если по доставке уже была успешная попытка.

SMTP-обработчик выбирает только queued attempts, у которых не отменена ни доставка, ни сама квитанция. Если квитанция отменена после успешной отправки, доставка остается в истории со статусом "Отменена после отправки", а успешная попытка остается как факт уже выполненного транспорта.

В dev по умолчанию используется `MAILER_DSN=null://null`, поэтому письмо считается отправленным, но наружу не уходит. Для production нужно задать реальный `MAILER_DSN`, `MAILER_FROM_EMAIL` и `MAILER_FROM_NAME`.

## Массовое Формирование

Для проведенного `billing_run` администратор может нажать "Сформировать квитанции" в карточке расчета. Система:

- берет участки с active posted начислениями этого расчета;
- создает по одной активной snapshot-квитанции на пару `billing_run` + участок;
- если активная квитанция уже есть, повторно ее не создает;
- для созданных и уже существующих квитанций ставит email-доставку в очередь по активным абонентам участка.

Связь хранится в `account_statements.billing_run_uuid`. Ручные квитанции из карточки участка могут оставаться без этой связи.

Карточка расчета показывает формирование квитанций как пятый этап сценария конца месяца. Статус считается как `active account_statements / участки с active posted начислениями`. Если квитанций еще нет, основная кнопка называется "Сформировать квитанции"; если сформированы не все - "Сформировать недостающие"; если сформированы все - вторичная "Обработать повторно". Повторная обработка не создает дубли квитанций, но может добавить недостающие доставки.

Доставки показываются отдельным транспортным шагом после формирования квитанций. Статус считается по активным `account_statement_deliveries` проведенного расчета и последней попытке доставки: `queued`, `sending`, `sent`, `failed`. Этот шаг не меняет финансовый статус расчета; он нужен, чтобы оператор видел, ушли ли письма, остались ли они в очереди или требуют внимания из-за ошибок.

Сформированные квитанции показываются отдельным блоком: участок, номер квитанции, сумму к оплате, дату создания, сводный статус доставок, ссылки на карточку квитанции и PDF.

Отдельный раздел `/admin/account-statement-deliveries` показывает очередь доставок по всему текущему хозяйству. Доступны фильтры по статусу: все, в очереди, отправляется, отправлено, ошибка, отменено. Поиск работает по номеру участка, номеру квитанции, email и имени получателя.

Отдельный раздел `/admin/account-statements` показывает сохраненные квитанции по всему текущему хозяйству. Доступны фильтры по номеру участка/квитанции, статусу квитанции, расчету, отсутствию расчета, дате формирования, сумме к оплате и наличию доставок. Из списка доступны быстрые переходы в карточку квитанции и PDF.
