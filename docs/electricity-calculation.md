# Расчет Электроэнергии

Документ фиксирует текущую модель расчета. Часть правил является предварительным решением и должна быть проверена на реальной эксплуатации.

## Источники Данных

Расчет строится из:

- показаний электросчетчиков;
- истории замены счетчиков;
- тарифных периодов;
- социальных норм;
- платежей;
- ручных корректировок.

## Показания

`ElectricityMeterReading` - это событие, а не месячная строка.

Система должна корректно обрабатывать ситуации:

- несколько показаний за месяц;
- пропущенные месяцы;
- показание, внесенное администратором за абонента;
- исправление ошибочного показания;
- замена счетчика.

## Расчетный Период

Для электроэнергии расчетный период - календарный месяц.

Дата формирования квитанций задается настройкой workspace. Предварительное значение - 5 число каждого месяца.

## Выбор Расчетного Показания

Показание считается актуальным для формирования квитанции, если дата фактического снятия попадает в окно актуальности перед датой формирования.

Предварительное правило:

```text
invoice_generation_day = 5
reading_freshness_window_days = 15
```

Если в окне актуальности есть несколько показаний, используется последнее по дате фактического снятия. Если дата фактического снятия одинаковая, используется последнее по времени отправки или внесения.

Если актуального показания нет, система должна поддержать два режима:

- рассчитать оценочное потребление по среднему предыдущему потреблению;
- пометить участок как требующий ручного снятия показаний и не формировать квитанцию автоматически.

Для MVP предпочтителен ручной режим: показать администратору список участков без актуальных показаний, чтобы оператор мог связаться с абонентом или самостоятельно снять показания.

Текущая реализация при создании draft `billing_run` проверяет каждую активную запись `account` текущего workspace и создает открытые записи в `billing_run_account_issues`. Для показаний проверяются активный электросчетчик участка, его тарифные зоны и последнее активное показание по каждой зоне на дату формирования. Если последнее показание есть, но дата снятия раньше начала окна актуальности, создается проблема `stale_reading`.

Кроме показаний, draft-проверка валидирует минимальную настройку расчета: действующее назначение тарифного профиля на начало расчетного периода, тарифный период этого профиля, глобальное правило диапазонов потребления для месяца расчетного периода, диапазоны правила и ставки тарифа для каждой пары `тарифная зона счетчика + диапазон потребления`. Текущая автоматическая проверка использует глобальное правило "все участки"; scope-aware выбор правил для групп и исключений в ней не применяется.

Повторная проверка draft `billing_run` пересобирает ожидаемый набор проблем. Если ранее открытая проблема больше не находится, она закрывается с причиной `resolved`. Если администратор вручную закрыл проблему с причиной `ignored`, повторная проверка не создает новую открытую проблему с тем же `account + issue_type` внутри того же `billing_run`.

Draft-начисление создается только для участка без открытых проблем в расчетном запуске. Для каждого регистра счетчика берется предыдущее активное показание на начало периода и текущее активное показание на дату формирования. Если пары показаний нет, начисление не создается, а в расчетный запуск добавляется проблема `calculation_error`.

Если администратор снял показание после даты формирования квитанций, оно может закрыть расчетный период только явным действием администратора:

- расчетный запуск должен быть в вычисляемом состоянии `draft`;
- участок должен иметь открытую проблему по отсутствующему или устаревшему показанию;
- выбранное позднее показание фиксируется в деталях начисления;
- действие попадает в audit log.

Если расчетный запуск уже опубликован, позднее показание не меняет начисление молча. Нужно создать пересчет: старое начисление переводится в `superseded`, новое начисление связывается с новым показанием.

## Потребление

В простом случае потребление считается как разница между текущим и предыдущим расчетным показанием:

```text
consumption_kwh = current_reading - previous_reading
```

При замене счетчика расчет должен учитывать конечное показание старого счетчика и начальное показание нового.

Если показания пропущены за один или несколько месяцев, система не должна молча создавать финансово значимые начисления без понятного правила. До отдельного решения такие периоды должны попадать в администраторский список контроля.

Показание меньше предыдущего не принимается как обычное показание. Допустимые сценарии:

- оформлена замена счетчика;
- администратор исправляет ошибочное предыдущее показание;
- администратор отменяет ошибочное новое показание.

Специальные случаи вроде переполнения счетчика не входят в MVP.

## Тарифы

Тарифная модель состоит из:

- тарифного профиля участка;
- периода действия тарифной сетки;
- тарифной зоны счетчика;
- диапазона потребления;
- ставки за 1 кВт*ч.

Система не хранит расписание тарифных зон внутри суток. Абонент передает готовые показания по регистрам счетчика, а расчет применяет ставку соответствующей зоны.

История тарифов хранится в системе. Если тариф был внесен неверно и исправлен задним числом, начисления за затронутые периоды должны пересчитываться.

Активные тарифные периоды не должны пересекаться внутри одного тарифного профиля. Если тариф исправляется задним числом, администратор должен указать основание изменения, а система должна пересчитать затронутые начисления.

## Социальные Нормы

`ElectricityConsumptionBandRule` задает, как потребление за месяц раскладывается по тарифным диапазонам.

Социальная норма является частным случаем такого правила:

```text
social_norm        0      150
above_social_norm  150    infinity
```

Система должна поддерживать:

- обычную месячную норму;
- повышенную норму для зимних месяцев;
- правила для всех участков;
- правила для групп участков;
- исключения для отдельных участков.

В MVP достаточно глобальных норм для всех участков. Группы и исключения остаются в модели, но могут не иметь отдельного UI в первом релизе.

Изменение правила диапазонов задним числом требует основания и пересчета затронутых начислений.

## Начисление

Начисление за период раскладывает потребление по тарифным зонам и диапазонам потребления.

Для MVP это может выглядеть как две строки для зоны `single`:

- `single + social_norm`;
- `single + above_social_norm`.

Для двухтарифного счетчика строк будет больше: например, `day + social_norm`, `day + above_social_norm`, `night + social_norm`, `night + above_social_norm`.

Сумма начисления:

```text
amount = sum(line_consumption_kwh * line_rate)
```

Для `total_proportional` общий расход по всем зонам сначала раскладывается по диапазонам, затем каждый диапазон распределяется между тарифными зонами пропорционально расходу зон. Для `per_tariff_zone` диапазоны применяются к расходу каждой зоны отдельно.

## Округление

Расчеты выполняются в точных decimal/numeric значениях.

Денежные компоненты начисления округляются до копеек по правилу half-up. Для положительных сумм это соответствует обычному округлению: 0,005 руб. округляется вверх.

```text
line_amount = round_money(line_consumption_kwh * line_rate)
amount = sum(line_amount)
```

Так итоговая сумма всегда совпадает с суммой отображаемых строк детализации. Правило округления должно быть единым в PHP и PostgreSQL и покрыто тестами.

## Пересчет

Пересчет требуется, если изменились:

- показания;
- тарифы;
- социальные нормы;
- счетчик или параметры его установки;
- ручные корректировки.

В MVP допустимо пересчитывать начисления синхронно для одного участка. Более сложный фоновый пересчет можно добавить позже.

Текущая генерация draft-начислений идемпотентна внутри одного `billing_run`: повторный запуск пропускает уже созданные активные начисления этого запуска. При каждом запуске обновляется `billing_run.accruals_generated_at`, даже если новых начислений не появилось.

Если во время генерации начислений возникла `calculation_error`, администратор может закрыть ее как `ignored`. Повторная генерация в том же `billing_run` не создает новую `calculation_error` для того же участка, чтобы ручное решение проблемы не превращалось в бесконечный цикл повторных ошибок.

Posting draft-расчета переводит все draft-начисления расчетного запуска в posted и фиксирует `billing_run.posted_at`. Posting не выполняется, если есть открытые `billing_run_account_issues`, если по расчетному запуску нет начислений или если после последнего запуска генерации менялись проблемы расчета.

## Квитанция

Сумма к оплате в квитанции - это текущий долг участка на дату формирования, а не сумма начисления только за последний месяц:

```text
amount_to_pay = sum(accruals_to_date) - sum(payments_to_date)
```

Если участок имеет переплату, квитанция на оплату не должна требовать повторную оплату уже закрытых начислений.
