# Абонентский Интерфейс

Документ фиксирует реализованный MVP личного кабинета абонента и ближайшие ограничения.

Статус: первый вертикальный сценарий реализован.

## Цель

Абонент должен войти в систему и без участия оператора:

- увидеть доступные ему хозяйства;
- выбрать текущее хозяйство для портала;
- увидеть свои участки;
- открыть карточку участка;
- передать новые показания активного электросчетчика;
- увидеть историю показаний, начислений, оплат и текущий баланс.

Портал не заменяет админку. Исправления, отмены, ручные начисления, внесение оплат, настройка тарифов и работа с проблемами расчетов остаются административными сценариями.

## Доступ

Абонентский доступ выводится из активной связи:

- `users.uuid = subscribers.user_uuid`;
- `subscribers.deleted_at is null`;
- `subscriber_account_accesses.revoked_at is null`;
- `accounts.deleted_at is null`.

Обычный абонент не имеет строки в `workspace_user_role_assignments`. Если пользователь одновременно оператор/администратор и абонент, он может видеть и админку, и личный кабинет.

В MVP любой активный `SubscriberAccountAccess` дает право смотреть участок и передавать показания. Значение `access_role` показывается пользователю, но не ограничивает действия. Разделение `owner`/`representative`/`viewer` на разные права оставляем на будущее.

## Контекст Хозяйства

Текущий `WorkspaceContext` сейчас обслуживает админку: глобальный администратор видит все хозяйства, а обычный пользователь видит хозяйства через роли `admin`/`operator`.

Для портала нужен отдельный контекст, потому что чистый абонент получает доступ не через `workspace_user_role_assignments`, а через `subscribers.user_uuid`.

Реализованное решение MVP:

- добавлен `SubscriberPortalContext`;
- доступные хозяйства портала получаются из активных `Subscriber`, привязанных к текущему `User`;
- текущий выбор портала хранится в отдельном session key `_snt_current_portal_workspace_uuid`;
- если выбранное хозяйство больше недоступно, выбирается первое доступное хозяйство портала;
- в UI портала использовать пользовательский термин "хозяйство".

Причина отдельного контекста: админский переключатель не должен показывать хозяйства, где у пользователя есть только абонентский доступ, иначе оператор сможет переключить админку на хозяйство, где у него нет административных прав.

Если позже понадобится строго единый выбор хозяйства на весь продукт, нужно будет выделить общий resolver с режимами `admin` и `portal`, чтобы каждый раздел корректно фильтровал доступные действия.

## Права

Реализованные security grants:

- `SUBSCRIBER_PORTAL_ACCESS` - пользователь имеет хотя бы одного активного `Subscriber` в текущем portal-хозяйстве;
- `SUBSCRIBER_ACCOUNT_VIEW` - пользователь имеет активный доступ к конкретному участку через `SubscriberAccountAccess`;
- `SUBSCRIBER_ACCOUNT_READING_SUBMIT` - в MVP совпадает с `SUBSCRIBER_ACCOUNT_VIEW`;
- `WORKSPACE_ACCESS` остается административным правом и не используется как пропуск в портал.

Портальные контроллеры должны всегда искать участок через пару "текущее portal-хозяйство + активный доступ абонента", а не просто по `account.uuid`.

## Страницы И Маршруты

### Dashboard

Маршрут: `GET /portal`

Назначение:

- стартовая страница личного кабинета;
- выбор текущего хозяйства, если у пользователя их несколько;
- список доступных участков;
- краткий баланс по каждому участку;
- быстрый переход к передаче показаний.

Пустое состояние:

- если у пользователя нет активных `Subscriber`, `SUBSCRIBER_PORTAL_ACCESS` не выдается и портал недоступен;
- если `Subscriber` есть, но нет активных участков, показывать сообщение, что участки еще не привязаны.

### Переключение Хозяйства

Маршрут: `POST /portal/workspaces/switch`

Действие:

- принимает `workspace_uuid`;
- проверяет, что выбранное хозяйство доступно пользователю как абоненту;
- сохраняет выбор в portal session key;
- возвращает пользователя на `/portal`.

### Список Участков

Маршрут: `GET /portal`.

В MVP отдельного маршрута `/portal/accounts` нет: главный экран личного кабинета и есть единый список участков абонента.

Колонки:

- номер участка;
- роль доступа;
- активный электросчетчик;
- баланс: долг, переплата или ноль.

### Карточка Участка

Маршрут: `GET /portal/accounts/{uuid}`

Показывает:

- номер участка;
- роль доступа пользователя к участку;
- текущий баланс;
- ссылку на экран "Баланс и операции";
- блок сохраненных квитанций, если система уже сформировала snapshot-документы;
- активный электросчетчик: модель, серийный номер, дата установки, дата поверки и дата окончания поверки;
- тарифные зоны активного счетчика;
- последние активные показания по зонам;
- история переданных показаний;
- история posted-начислений;
- история оплат.

Отдельные административные данные вроде audit log, внутренних uuid и причин отмен показывать не нужно.

### Передача Показаний

Маршрут: `GET|POST /portal/accounts/{uuid}/readings/new`

Форма:

- дата снятия показаний, по умолчанию сегодня в timezone текущего portal-хозяйства;
- по одному числовому полю на каждый регистр активного счетчика;
- комментарий абонента опционально.

Правила:

- передавать показания можно только для активного счетчика участка;
- если активного счетчика нет, форма недоступна;
- если у счетчика несколько тарифных зон, форма показывает все зоны;
- все зоны активного счетчика обязательны к заполнению;
- дата показаний не должна быть в будущем;
- дата показаний не должна быть раньше даты установки счетчика;
- дата показаний не должна быть позже даты снятия счетчика, если счетчик снят;
- новое значение по каждой зоне не должно быть меньше предыдущего активного показания этой зоны;
- новое значение по каждой зоне не должно быть больше следующего активного показания этой зоны, если такое показание уже есть;
- правила дат, наличия регистра и монотонности проверяются общим сервисом `ElectricityMeterReadingValidator`, который используется и в админке, и в портале;
- перед вводом показывается последнее активное показание по каждой зоне;
- при успешной отправке создаются `ElectricityMeterReading` со `source = subscriber`, `submitted_by = current User`, `provided_by_subscriber = current Subscriber`;
- каждое созданное показание пишется в audit log;
- после submit redirect идет на карточку участка.

Если часть зон не заполнена, в MVP лучше считать форму невалидной. Частичная передача по многотарифному счетчику может приводить к неполному расчету и должна проектироваться отдельно.

### История Показаний

Маршрут: `GET /portal/accounts/{uuid}/readings`

В первом MVP отдельный маршрут не добавлен: история показаний показывается на карточке участка `GET /portal/accounts/{uuid}` с серверной пагинацией и фильтрами по статусу и дате снятия.

Показывается:

- дата снятия;
- зона;
- значение;
- дата отправки;
- источник;
- статус: активное, заменено, отменено.

В MVP отмененные и замененные записи можно показывать в истории, но визуально отделять от активных. Действий исправления у абонента нет.

### Начисления И Оплаты

Маршруты:

- `GET /portal/accounts/{uuid}/accruals`;
- `GET /portal/accounts/{uuid}/payments`.

В первом MVP отдельные маршруты не добавлены: блоки показываются на карточке участка с независимой серверной пагинацией и фильтрами по статусу и датам.

Начисления:

- показывать posted начисления;
- draft-записи не показывать абоненту;
- cancelled и superseded записи можно показывать только как исторические, но не учитывать в балансе;
- для электроэнергии показывать период, сумму и строки расчета по социальной норме/сверх нормы, если контекст расчета есть.

Оплаты:

- показывать оплаты участка;
- cancelled и superseded записи можно показывать только как исторические, но не учитывать в балансе;
- дату, сумму, плательщика и назначение платежа показывать, если они заполнены.

Пагинация историй независимая: у показаний, начислений и оплат отдельные query-параметры страницы, поэтому переход по страницам одного блока не сбрасывает положение других блоков.

### Баланс И Операции

Маршрут: `GET /portal/accounts/{uuid}/balance`.

Показывает текущую read-only сверку по участку:

- итог active posted начислений;
- итог active оплат;
- текущий баланс;
- сумму к оплате или переплату;
- список начислений и оплат;
- детализацию расчета электроэнергии, если она есть.

Экран не является сохраненной квитанцией, не имеет номера документа, PDF и QR-кода. Это текущий расчетный срез по данным системы.

### Квитанции

Маршруты:

- `GET /portal/accounts/{uuid}/statements/{statementUuid}` - просмотр сохраненной квитанции по доступному участку;
- `GET /portal/accounts/{uuid}/statements/{statementUuid}/pdf` - PDF сохраненной квитанции.

Квитанции показываются в отдельном блоке карточки участка и читаются только из active `account_statements`. Отмененные snapshot-документы абоненту не показываются.

Если у участка есть долг, но сохраненной квитанции еще нет, портал прямо пишет, что задолженность есть, а квитанция для оплаты еще не сформирована. Это нужно, чтобы не смешивать баланс и платежный документ.

Подробнее: [account-statements.md](account-statements.md).

## Навигация

После login:

- если пароль протух, сначала работает существующий forced password flow;
- если пользователь имеет только абонентский доступ, корневой экран должен вести его в `/portal`;
- если пользователь имеет административный доступ, показывать ссылку в админку;
- если пользователь одновременно абонент и администратор/оператор, показывать обе ссылки: "Личный кабинет" и "Админка".

В портале нужен отдельный shell:

- сверху: название "КомУчёт", текущий пользователь, выбор хозяйства, профиль, выход;
- навигация содержит `Главная` и `Участки`;
- основной раздел: участки и действия абонента;
- без левого административного меню.

В списках участков нужно давать прямое действие "Передать показания" для участка с активным электросчетчиком. Это основной регулярный сценарий абонента, поэтому он не должен требовать обязательного захода в карточку участка.

## Реализованный Сценарий

Минимальный вертикальный сценарий:

1. Пользователь входит по email и паролю.
2. Система определяет portal-хозяйство через активный `Subscriber`.
3. `/portal` показывает список участков из активных `SubscriberAccountAccess`.
4. Абонент открывает участок.
5. Система показывает активный электросчетчик, текущий баланс и последние показания.
6. Абонент открывает форму передачи показаний.
7. Абонент вводит дату и значения по зонам.
8. Система валидирует доступ, счетчик, дату и монотонность показаний.
9. Система сохраняет `ElectricityMeterReading`.
10. Абонент возвращается на карточку участка и видит новое показание в истории.

Сценарий закреплен functional smoke-тестом `SubscriberPortalControllerTest::testSubscriberCanCompletePortalFlowThroughLogin`. Тест проходит через реальную форму `/login`, редирект чистого абонента в `/portal`, dashboard, карточку участка, экран "Баланс и операции" и отправку нового показания.

3. Готово: сделан shell и dashboard `/portal`.
4. Готово: сделаны список и карточка участков с балансом и активным счетчиком.
5. Готово: добавлена read-only история показаний, начислений и оплат на карточке участка.

7. Готово: добавлены functional-тесты сценариев абонента и smoke-проход через реальный login.

## Ближайшие Улучшения После Первого Шага

- при необходимости вынести истории в отдельные маршруты `/portal/accounts/{uuid}/readings`, `/portal/accounts/{uuid}/accruals`, `/portal/accounts/{uuid}/payments`;
- позже разделить права `owner`/`representative`/`viewer`, если появится сценарий просмотра без права передачи показаний.

## Не Входит В Первый Портальный Шаг

- самостоятельная регистрация;
- смена email и подтверждение email;
- передача показаний без активного счетчика;
- исправление или отмена показаний абонентом;
- онлайн-оплата;
- скачивание PDF-квитанции;
- уведомления;
- мобильное приложение или SPA.
