# Демо-Данные

Документ фиксирует план реализации demo seed CLI для обучения операторов, демонстраций и проверки pilot-сценариев.

## Цель

Нужно уметь создать отдельное демо-хозяйство с псевдоданными, которые сразу показывают основные возможности КомУчёта:

- работу администратора и оператора хозяйства;
- личный кабинет абонента;
- участки, абонентов, представителей и доступы;
- электросчетчики, тарифы, показания и начисления;
- оплаты, балансы, квитанции, PDF/QR и email-доставки;
- проблемные ситуации, которые оператор должен уметь разбирать.

Это не тестовые fixtures. Команда должна быть продуктовым CLI-инструментом, который можно запускать в dev, staging или на демонстрационном production-стенде при явном подтверждении.

## CLI

Базовая команда:

```bash
php bin/console app:demo:seed
```

В Docker dev-контуре:

```bash
make demo-seed ARGS="--size=small --as-of=2026-05-14"
```

Планируемые опции:

```bash
php bin/console app:demo:seed \
    --workspace-code=demo \
    --workspace-name="Демо-хозяйство КомУчёт" \
    --size=medium \
    --as-of=2026-05-14 \
    --seed=komuchet-demo \
    --grant-admin-email=admin@example.test \
    --grant-operator-email=operator@example.test \
    --grant-subscriber-email=subscriber@example.test \
    --reset \
    --confirm=demo
```

Опции:

- `--workspace-code` - код создаваемого или обновляемого демо-хозяйства. По умолчанию `demo`.
- `--workspace-name` - отображаемое название хозяйства.
- `--size` - объем данных: `small`, `medium`, `large`.
- `--as-of` - опорная дата генерации. От нее строятся последние 12 месяцев показаний, оплат и расчетов.
- `--seed` - seed детерминированной генерации. Один и тот же seed и `as-of` должны давать одинаковые данные.
- `--grant-admin-email` - выдать роль `admin` в демо-хозяйстве существующему пользователю. Можно указывать несколько раз.
- `--grant-operator-email` - выдать роль `operator` в демо-хозяйстве существующему пользователю. Можно указывать несколько раз.
- `--grant-subscriber-email` - связать существующего пользователя с демо-абонентом для проверки личного кабинета. Можно указывать несколько раз.
- `--reset` - удалить и пересоздать данные демо-хозяйства перед генерацией.
- `--confirm=demo` - обязательное подтверждение для destructive reset.
- `--allow-prod-reset` - дополнительное явное подтверждение для reset в production-like окружении.

Команда должна печатать итоговую сводку:

- UUID и код хозяйства;
- список существующих пользователей, которым выдан доступ к демо-хозяйству;
- список существующих пользователей, которым выдан доступ к личному кабинету через связь с демо-абонентом;
- число созданных участков, абонентов, счетчиков, показаний, оплат, начислений, квитанций и доставок;
- список специально созданных сценариев.

## Размеры Наборов

`small`:

- 15-20 участков;
- 10-15 абонентов;
- 2-3 расчетных месяца;
- подходит для быстрой smoke-демонстрации.

`medium`:

- 40-50 участков;
- 30-40 абонентов;
- 12 месяцев истории;
- основной режим для обучения оператора.

`large`:

- 80-100 участков;
- 60-80 абонентов;
- 12-18 месяцев истории;
- нужен для проверки списков, фильтров, поиска и производительности интерфейса.

## Утвержденный Состав Данных

Минимальный полезный набор:

- доступ к демо-хозяйству для существующих пользователей через роли `admin` и `operator`;
- демо-абоненты и возможность выдать существующим пользователям доступ в портал через `--grant-subscriber-email`;
- 30-80 участков, в зависимости от `--size`;
- владельцы, супруги/представители, случаи "один человек владеет несколькими участками";
- группы участков: `Летние`, `Круглогодичные`;
- однотарифные и двухтарифные электросчетчики;
- модели, серийные номера, даты поверки;
- показания за последние 12 месяцев;
- оплаты с разными сценариями: долг, переплата, ровный баланс, пропущенная оплата;
- тарифные зоны, тарифные профили, социальные нормы;
- несколько проведенных `billing_run`;
- сохраненные квитанции, готовые к PDF/QR;
- доставки квитанций в разных состояниях: queued, sent, failed, cancelled;
- несколько открытых `billing_run_account_issues`, чтобы оператор видел проблемные кейсы.

Данные должны быть сценарными, а не просто случайными.

Обязательные сценарии:

- участок с долгом;
- участок с переплатой;
- участок без актуальных показаний;
- участок с исправленным показанием;
- участок с двумя владельцами или представителями;
- абонент с двумя участками;
- двухтарифный счетчик;
- квитанция в очереди на отправку;
- квитанция с неуспешной доставкой.

## Правила Данных

- Все генерируемые email демо-абонентов должны использовать reserved-домен `example.test`.
- Email существующих пользователей в `--grant-admin-email` и `--grant-operator-email` могут быть реальными, потому что эти пользователи уже заведены в системе.
- ФИО, телефоны, номера участков, серийные номера и банковские реквизиты должны быть полностью вымышленными.
- Для платежных реквизитов используются демонстрационные значения, непригодные для реальных платежей.
- Суммы, даты и потребление должны выглядеть реалистично, но не должны копировать реальные данные.
- В демо-данных допустимы специальные подсказки в названиях, например `9-003 долг`, только если они не ломают реалистичность интерфейса. Предпочтительно показывать сценарий через состояние данных, а не через искусственное название.

Пользователи для входа в админку не создаются demo seed автоматически. Команда только выдает роли демо-хозяйства уже существующим пользователям с активным подтвержденным email. Если email неизвестен системе, команда должна завершиться ошибкой до записи демо-данных.

Портальные учетные записи демо-абонентов также не создаются автоматически. Для smoke-проверки личного кабинета команда может связать уже существующего пользователя с демо-абонентом через `--grant-subscriber-email`. Первый такой пользователь получает владельца участка `9-001`, у которого в small-наборе есть доступ к участкам `9-001` и `9-013`.

## Генерация Людей

`DemoPeopleCatalog` должен уметь детерминированно создать не менее 500 псевдоперсон.

Исходные списки:

- 30 распространенных русских фамилий;
- 30 распространенных мужских имен;
- 30 распространенных женских имен;
- 30 распространенных мужских отчеств;
- 30 распространенных женских отчеств.

ФИО генерируются вперемешку по полу, но без грамматически неверных сочетаний:

- мужская запись: мужская форма фамилии + мужское имя + мужское отчество;
- женская запись: женская форма фамилии + женское имя + женское отчество;
- семейная пара: общая фамилия в мужской и женской форме.

Email генерируется не из ФИО, а из стабильного demo-id:

```text
demo.subscriber.001@example.test
demo.subscriber.002@example.test
```

Это уменьшает риск случайного совпадения с реальным адресом и сохраняет детерминированность.

## Безопасность Reset

`--reset` должен быть ограничен:

- разрешено сбрасывать только хозяйства с кодом `demo` или `demo-*`;
- требуется `--confirm=demo`;
- в production-like окружении нужен дополнительный явный флаг, например `--allow-prod-reset`;
- глобальные пользователи, email, пароли и другие identity-данные не удаляются и не изменяются;
- workspace-роли существующих пользователей удаляются только в рамках сбрасываемого демо-хозяйства и затем могут быть выданы заново через `--grant-admin-email` или `--grant-operator-email`.

Предпочтительный подход: команда не должна делать raw delete по всей базе. Она должна работать в пределах одного `workspace_uuid` и уважать workspace-изоляцию.

## Архитектура

Планируемые классы:

```text
src/Command/DemoSeedCommand.php
src/Demo/DemoDataSeeder.php
src/Demo/DemoDatasetSize.php
src/Demo/DemoScenarioCatalog.php
src/Demo/DemoPeopleCatalog.php
src/Demo/DemoElectricityProfileGenerator.php
src/Demo/DemoPaymentScenarioGenerator.php
src/Demo/DemoStatementScenarioGenerator.php
```

Ответственность:

- `DemoSeedCommand` - CLI-интерфейс, валидация опций, подтверждения, итоговый вывод.
- `DemoDataSeeder` - orchestration всего сценария и транзакционные границы.
- `DemoDatasetSize` - параметры `small`, `medium`, `large`.
- `DemoScenarioCatalog` - список обязательных бизнес-сценариев и распределение их по участкам.
- `DemoPeopleCatalog` - детерминированные псевдо-ФИО и email; каталог должен выдавать не менее 500 псевдоперсон на базе списков распространенных фамилий, имен и отчеств. Статус: реализовано.
- `DemoElectricityProfileGenerator` - счетчики, регистры, тарифы, показания, расчетные периоды.
- `DemoPaymentScenarioGenerator` - оплаты и итоговые финансовые состояния.
- `DemoStatementScenarioGenerator` - сохраненные квитанции и доставки.

Если существующие доменные сервисы уже умеют выполнять нужное действие, demo seed должен использовать их, а не дублировать бизнес-логику:

- генерация начислений через `ElectricityBillingRunAccrualGenerator`;
- генерация проблем расчета через `BillingRunIssueGenerator`;
- snapshot квитанций через `BillingRunStatementGenerator` или `AccountStatementSnapshotGenerator`;
- постановка доставок через `AccountStatementDeliveryEnqueuer`;
- PDF и QR не нужно заранее сохранять, если они строятся из snapshot по запросу.

Текущая реализация создает один демонстрационный профиль платежных реквизитов `demo-default` и активное назначение по умолчанию для хозяйства. Реквизиты копируются в snapshot-квитанции при генерации, а QR-код строится динамически из snapshot-реквизитов при открытии печатной формы или PDF. При повторном запуске команда дополняет реквизитами уже существующие демо-snapshot, если они были созданы до появления реквизитов.

## Детерминированность

Команда должна использовать deterministic random source:

- один `--seed`;
- одна `--as-of` дата;
- одинаковая версия генератора;
- одинаковый размер набора.

Это нужно, чтобы при демонстрациях и тестировании можно было воспроизводить одни и те же кейсы.

## Реализация По Шагам

1. Сделать `DemoSeedCommand` с выводом параметров, валидацией `--size`, `--as-of`, `--reset`, `--confirm`. Статус: реализовано.
2. Добавить создание или обновление демо-хозяйства и `billing_settings`. Статус: реализовано.
3. Выдать доступ к демо-хозяйству существующим пользователям через `--grant-admin-email` и `--grant-operator-email`. Статус: реализовано.
4. Создать тарифные зоны, тарифные профили, тарифные периоды, ставки и правила социальных норм. Статус: реализовано.
5. Создать группы `Летние` и `Круглогодичные`. Статус: реализовано.
6. Создать участки, абонентов, представителей и доступы абонентов к участкам. Статус: реализовано.
   Портальные учетные записи для демо-абонентов не создаются, но существующего пользователя можно связать с демо-абонентом через `--grant-subscriber-email`.
7. Создать электросчетчики, регистры и истории показаний.
   Статус: реализовано - созданы электросчетчики, регистры, назначения тарифных профилей и 12 месяцев показаний. Есть сценарии двухтарифного счетчика, устаревших показаний и исправленного показания.
8. Создать оплаты и финансовые сценарии.
   Статус: реализовано - создано posted-начисление по каждому участку и оплаты для сценариев долг, переплата, ровный баланс и пропущенная оплата.
9. Создать несколько `billing_run`, провести часть из них и оставить несколько проблем.
   Статус: реализовано - создаются три расчета: два прошлых расчета проводятся, текущий остается черновиком с открытой проблемой по устаревшим показаниям.
10. Сформировать snapshot-квитанции и доставки в разных состояниях.
    Статус: реализовано - по проведенным расчетам создаются snapshot-квитанции и доставки в состояниях queued, sent, failed и cancelled.
11. Добавить фактическое выполнение `--reset` с защитами.
    Статус: реализовано - удаляются только workspace-scoped данные хозяйства `demo`/`demo-*`, глобальные пользователи сохраняются, после reset хозяйство создается заново.
12. Добавить functional-тест команды на `small` dataset. Статус: реализовано.
13. Добавить smoke-проверку: как запустить, какими логинами входить, какие сценарии смотреть.
    Статус: реализовано через `make e2e-smoke-full`.

## Тестирование

Минимальные проверки:

- команда `app:demo:seed --size=small` успешно проходит на чистой test-БД;
- повторный запуск без `--reset` не ломает данные и дает понятное сообщение;
- `--reset --confirm=demo` пересоздает только demo-хозяйство;
- демо-оператор может войти в `/admin`;
- демо-абонент может войти в `/portal`;
- хотя бы один участок имеет долг, один переплату, один неактуальные показания, один двухтарифный счетчик;
- есть хотя бы одна сохраненная квитанция и одна доставка в статусах queued, sent, failed, cancelled.

## Не Входит В Первый Шаг

- генерация реалистичных юридических документов;
- импорт внешних CSV/Excel как источник демо-данных;
- UI-кнопка "создать демо-хозяйство";
- отправка реальных email при генерации демо-данных;
- сохранение PDF-файлов на диск.
