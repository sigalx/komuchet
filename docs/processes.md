# Бизнес-Процессы

## Текущий Ручной Процесс

### 1. Сбор Показаний

Жители отправляют показания электросчетчиков на корпоративный почтовый ящик СНТ.

Оператор:

- открывает почтовый ящик;
- читает письма;
- определяет участок и абонента;
- переносит показания в Excel или аналогичную таблицу.

### 2. Учет Оплат

Бухгалтер периодически передает оператору текстовую сводку поступлений на банковский счет СНТ.

Ожидаемый состав строки поступления:

- кто оплатил;
- дата;
- сумма;
- комментарий или назначение платежа.

Оператор вручную сопоставляет поступления с участками.

### 3. Формирование Рассылки

Оператор формирует для абонента отчет и квитанцию, затем отправляет их по электронной почте.

PDF из корня проекта показывает текущий вид результата: акт сверки с начислениями, оплатами, долгом и платежной квитанцией.

## Целевое Состояние

### Этап 1

Система заменяет ручной прием показаний через почту.

Абонент самостоятельно передает показания через личный кабинет. Администратор может делать то же самое за абонента.

Перед формированием квитанций система показывает администратору участки без актуальных показаний. Администратор может связаться с абонентом или самостоятельно снять показания.

Администратор задает настройки биллинга для хозяйства: название хозяйства в квитанциях, день формирования квитанций, часовой пояс и окно актуальности показаний.

Расчет электроэнергии начинается с `billing_run`: администратор создает черновик расчета за период. При создании черновика система уже генерирует рабочий список проблем по участкам в `billing_run_account_issues`: нет активного счетчика или тарифных зон, нет актуальных показаний, показания устарели, не назначен тарифный профиль, нет тарифного периода, нет глобального правила нормы, не заданы диапазоны нормы или ставки тарифа.

`billing_run` не создает финансовые начисления автоматически. Начисления и posting выполняются отдельными действиями после проверки найденных проблем.

В draft-расчете администратор может запустить повторную проверку. Повторная проверка обновляет открытые проблемы, создает новые найденные проблемы и автоматически закрывает как `resolved` те открытые проблемы, которые больше не воспроизводятся. Проблема, вручную закрытая как `ignored`, не открывается повторно для того же участка, типа проблемы и расчетного запуска.

Отдельный админский экран показывает все открытые `billing_run_account_issues` текущего хозяйства, чтобы оператор видел общий рабочий список без перехода в каждый расчет отдельно. Список фильтруется по расчету, участку и типу проблемы. Оператор может закрыть проблему прямо из этого списка; закрытие сохраняет выбранные фильтры и доступно только для draft-расчетов.

Пока в draft-расчете есть открытые проблемы, проведение расчета считается заблокированным. Финансовый posting будет обязан проверять отсутствие открытых `billing_run_account_issues`.

Draft-начисления электроэнергии генерируются отдельным действием администратора из карточки `billing_run`. Система создает начисления только по участкам без открытых проблем и не создает дубль, если начисление для участка в этом расчетном запуске уже есть. Участки с открытыми проблемами пропускаются до исправления или ручного закрытия проблем. Каждый запуск генерации фиксируется в `billing_run.accruals_generated_at`.

После генерации администратор проводит draft-расчет. Posting запрещен, если в `billing_run` остаются открытые проблемы, если начисления еще не сгенерированы или если после последней генерации менялись `billing_run_account_issues`. Успешный posting проставляет `posted_at` у `billing_run` и у всех draft-начислений этого запуска. С этого момента начисления участвуют в вычисляемом балансе участка.

### Этап 2

Система поддерживает ручной учет оплат на основе данных от бухгалтера.

Администратор вносит оплаты в интерфейсе. Баланс участка пересчитывается автоматически.

Система должна учитывать, что сумма к оплате в квитанции является текущим долгом участка, а не только начислением за последний месяц.

### Этап 3

Система принимает бухгалтерскую сводку как файл или текст, разбирает ее и предлагает сопоставления оплат с участками.

Автоматическое сопоставление должно быть проверяемым: спорные строки остаются на ручное подтверждение.

### Этап 4

Система формирует отчеты и квитанции автоматически и отправляет их абонентам.

Формат отчета должен быть сопоставим с текущим PDF, но не обязан полностью повторять Excel-верстку.

### Ручные Начисления

MVP поддерживает ручные начисления по участку:

- годовой членский взнос;
- годовая фиксированная плата за воду при наличии подключения;
- целевые или разовые платежи по решениям СНТ.

Администратор вводит сумму, период, тип и комментарий. Формализация расчетных правил по площади участка, подключению воды и другим параметрам является отдельным дальним этапом.
