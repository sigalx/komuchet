# Импорт Квитанций Заветы Мичурина

Документ описывает custom-компонент для разбора PDF-квитанций СНТ "Заветы Мичурина".
Это не часть универсальной бизнес-логики КомУчета, а адаптер под конкретный исторический формат документов одного хозяйства.

## Цель

Оператор может загрузить накопленные PDF-квитанции, которые раньше рассылались садоводам вручную, и получить из них структурированные данные:

- участок;
- ФИО владельца из исторической квитанции;
- электросчетчик;
- помесячные показания и начисления;
- платежи, напечатанные в акте сверки;
- итоговые суммы;
- платежные реквизиты из формы ПД-4.

Первый шаг реализован как dry-run parser без записи в базу. Это нужно, чтобы проверить качество распознавания на реальных обезличенных примерах до применения данных к доменным сущностям.

Второй шаг добавляет staging-таблицы: можно сохранить пачку разбора и JSON-результат по каждому файлу, но участки, абоненты, платежи, показания и начисления все еще не создаются автоматически.

Третий шаг добавляет read-only предпросмотр применения. Он сопоставляет один распознанный файл с уже существующими сущностями хозяйства и показывает оператору, какие данные будут переиспользованы, какие нужно создать, а какие требуют ручной проверки. Основные таблицы на этом шаге не меняются.

Четвертый шаг добавляет ручное применение одного распознанного файла. Данные записываются в доменную модель только после подтверждения оператором в карточке файла.

## Граница Компонента

Код лежит отдельно:

- `src/Custom/ZavetyMichurina/ElectricityStatementImport/` - parser, DTO и извлечение текста из PDF;
- `src/Custom/ZavetyMichurina/ElectricityStatementImport/ZavetyMichurinaStatementImportPreviewBuilder.php` - read-only предпросмотр применения staging-файла;
- `src/Custom/ZavetyMichurina/ElectricityStatementImport/ZavetyMichurinaStatementImportApplier.php` - подтвержденное применение staging-файла к доменным сущностям;
- `src/Command/ZavetyMichurinaParseElectricityStatementCommand.php` - CLI-команда dry-run.
- `src/Command/ZavetyMichurinaStageElectricityStatementsCommand.php` - CLI-команда сохранения результата в staging.
- `zavety_michurina_statement_import_batches` - пачки загрузки.
- `zavety_michurina_statement_import_files` - отдельные файлы и результат разбора.

Компонент не должен попадать в универсальные сервисы расчета, квитанций или платежей. Когда появится UI/импорт, общая система должна видеть уже подтвержденные доменные операции, а не особенности PDF-формата конкретного СНТ.

## Источник Истины

PDF-квитанция "Заветы Мичурина" не является источником истины для квитанций КомУчета.

Импорт из PDF может создавать или сопоставлять только исходные доменные данные:

- участки и абонентов;
- связи абонентов с участками;
- электросчетчики и их показания;
- тарифы, социальные нормы и исторические начисления;
- платежи, напечатанные в акте сверки;
- платежные реквизиты как справочные данные.

Сохраненные `account_statements` не должны создаваться напрямую из импортированной PDF-квитанции. Квитанции КомУчета формируются только самой системой из подтвержденных доменных данных: расчетов, posted-начислений, оплат, платежных реквизитов и вычисленного баланса. Поэтому импортированный PDF может помочь восстановить историю, но не подменяет системный расчет и системную snapshot-квитанцию.

Для контроля качества импорта нужна сверка остатка к оплате: система должна пересчитать `начислено - оплачено` по импортированным строкам и сравнить результат с итоговой суммой, напечатанной в PDF. Если значения не сходятся, импорт должен явно показать это оператору в предпросмотре или результате применения, а не молча считать данные корректными.

## CLI Dry-Run

Команда:

```bash
php bin/console app:zm:parse-electricity-statement docs/samples/electricity-statement-redacted.pdf
```

Команда печатает JSON и не меняет базу.

Для тестов и отладки без `pdftotext` можно передать уже извлеченный layout-текст:

```bash
php bin/console app:zm:parse-electricity-statement tests/Fixtures/zavety_michurina/electricity-statement-layout.txt --from-text
```

## CLI Staging

Команда:

```bash
php bin/console app:zm:stage-electricity-statements --workspace=demo --name="Исторические PDF" docs/samples/electricity-statement-redacted.pdf
```

Команда создает:

- один import batch;
- по одной file-записи на каждый входной файл;
- `parsed_result jsonb` с dry-run JSON;
- `status = parsed` или `status = failed`.

Для тестового layout-текста:

```bash
php bin/console app:zm:stage-electricity-statements --workspace=demo --from-text tests/Fixtures/zavety_michurina/electricity-statement-layout.txt
```

Для исторической загрузки директории PDF:

```bash
php bin/console app:zm:stage-electricity-statements --workspace=demo --name="Исторические PDF" /path/to/statements
```

Команда принимает и файлы, и директории. Для директорий по умолчанию берутся `*.pdf`, а с `--from-text` - `*.txt`.
Дополнительные параметры:

- `--pattern="*.pdf"` - изменить маску файлов при сканировании директорий;
- `--recursive` - искать файлы во вложенных директориях.

При разработке под Docker CLI-команда выполняется внутри PHP-контейнера и видит только bind-mounted каталоги проекта.
Для локальной массовой загрузки реальные PDF нужно складывать в игнорируемый каталог проекта, например:

```bash
mkdir -p local/imports/zavety-michurina
cp ~/Downloads/*.pdf local/imports/zavety-michurina/
make zm-stage ZM_WORKSPACE=demo ZM_IMPORT_PATH=local/imports/zavety-michurina ZM_IMPORT_NAME="Исторические PDF"
```

Каталог `local/` добавлен в `.gitignore`; оригинальные PDF с персональными данными не должны попадать в Git.
Каталог `var/` для этого не подходит: в dev-контейнере он перекрыт Docker volume и не совпадает с `var/` на хосте.

Makefile-обертка поддерживает параметры:

- `ZM_WORKSPACE=demo` - код хозяйства;
- `ZM_IMPORT_PATH=local/imports/zavety-michurina` - файл или директория для staging;
- `ZM_IMPORT_NAME="Исторические PDF"` - название пачки;
- `ZM_PATTERN="*.pdf"` - маска файлов для директории;
- `ZM_RECURSIVE=1` - искать во вложенных директориях;
- `ZM_FROM_TEXT=1` - импортировать заранее извлеченные `pdftotext -layout` файлы.

Даже успешный staging не применяет данные к рабочей модели. Это только безопасная промежуточная зона.

## CLI Apply

После проверки staging-пачки ее можно применить из CLI:

```bash
php bin/console app:zm:apply-statement-import-batch --workspace=demo <batch-uuid>
```

Через Makefile:

```bash
make zm-apply ZM_WORKSPACE=demo ZM_BATCH=<batch-uuid>
```

Команда применяет только файлы в статусе `parsed` и использует тот же applier, что web-кнопки. Полностью примененные файлы переходят в `applied`, частичные остаются в `parsed`.

По умолчанию техническая ошибка останавливает команду. Если нужно продолжить остальные файлы после ошибки:

```bash
php bin/console app:zm:apply-statement-import-batch --workspace=demo --continue-on-error <batch-uuid>
```

Через Makefile:

```bash
make zm-apply ZM_WORKSPACE=demo ZM_BATCH=<batch-uuid> ZM_CONTINUE_ON_ERROR=1
```

Exit code:

- `0` - все найденные parsed-файлы применены без пропусков, либо parsed-файлов в пачке уже нет;
- `1` - были технические ошибки или частично примененные файлы, требующие ручного разбора;
- `2` - неверные аргументы команды.

## Административный Просмотр

Экран доступен в админке:

- `/admin/zavety-michurina/statement-imports` - список пачек импорта текущего хозяйства;
- `/admin/zavety-michurina/statement-imports/new` - загрузка PDF в новую staging-пачку;
- карточка пачки - список файлов, статусы, распознанный участок и ФИО, отчет применения, массовое применение распознанных файлов;
- карточка файла - сводка распознанных данных, read-only предпросмотр применения, ошибка разбора или JSON dry-run результата;
- удалить staging-файл можно только до применения. Примененный файл не удаляется, потому что по нему уже созданы доменные данные и audit trail.

При web-загрузке оригиналы PDF пока не сохраняются в файловое хранилище. В базе остаются имя файла, SHA-256, размер и результат распознавания. Это снижает риск случайного хранения персональных данных до отдельного решения по приватному storage.

Экран предназначен для контроля качества разбора и ручного применения одного файла. Кнопка `Применить` доступна только для файлов в статусе `parsed`, если предпросмотр не нашел блокирующих проблем.

Предпросмотр применения показывает:

- участок: найден активный участок по номеру или потребуется создание;
- абонента: найден активный абонент по ФИО или потребуется создание;
- связь абонента с участком: найдена активная связь или потребуется создание;
- электросчетчик: найден активный счетчик, потребуется создание или нужна ручная проверка;
- сегменты счетчика: если показания в PDF снизились, это выводится как отдельные сегменты; если в PDF явно указан новый счетчик, предпросмотр показывает это как подтвержденную замену, а не как предупреждение;
- тарифную зону: для PDF нужен один целевой register, потому что сам PDF содержит готовые суммарные показания без внутрисуточных зон;
- тарифы: уникальные непрерывные наборы ставок по соцнорме и сверх нормы, чтобы при применении не плодить дубли;
- соцнормы: значения по расчетным периодам `год + месяц`; разные нормы одного календарного месяца в разные годы считаются нормальной ситуацией;
- показания: количество строк, а при однозначно найденном счетчике и зоне - новые строки, дубли и конфликты; дата показания берется как первое число месяца, следующего за расчетным периодом строки PDF;
- оплаты: строки оплат, дедуплицируемые по участку, дате оплаты и сумме;
- платежные реквизиты: существующий профиль реквизитов или необходимость создать/сопоставить новый;
- начисления и остаток к оплате: исторические строки начислений, итоговые суммы из PDF и сверка пересчитанного остатка с напечатанной итоговой суммой.

Предпросмотр не является apply-операцией и не должен менять доменные таблицы. Его задача - сделать будущую apply-операцию предсказуемой и проверяемой.

## Применение Файла

Apply-операция выполняется из карточки файла и работает в одной транзакции. При успешном применении создаются или переиспользуются:

- участок по номеру из PDF;
- абонент по ФИО из PDF;
- связь абонента с участком с ролью владельца;
- профиль платежных реквизитов, если реквизиты еще не заведены;
- назначение импортированных реквизитов для начислений по электроэнергии, если для электроэнергии еще нет активного назначения;
- базовые справочники для однотарифного импорта: тарифная зона `single`, тарифный профиль `imported_electricity`, группы потребления `social_norm` и `above_social_norm`;
- назначение тарифного профиля импортируемому участку;
- тарифные периоды и ставки по соцнорме/сверх нормы;
- правила социальной нормы по месяцам и диапазоны потребления;
- электросчетчики и registers;
- показания электросчетчика;
- оплаты;
- исторические posted-начисления по электроэнергии.

Apply-операция не создает snapshot-квитанции. После импорта квитанции должны формироваться штатным механизмом КомУчета, чтобы источником истины были собственные расчеты системы.

Дедупликация сделана по устойчивым бизнес-ключам: участок по номеру, абонент по ФИО, тарифный профиль по коду, тарифные периоды по профилю и `valid_from`, ставки по периоду/зоне/диапазону, правила соцнорм по профилю/месяцу/`valid_from`, оплаты по участку/дате/сумме, начисления по участку/типу/периоду, показания по счетчику/зоне/дате.

Строка PDF вида `2026 апрель` считается расчетом за апрель: начисление получает период `2026-04-01` - `2026-05-01`, а показание счетчика из этой строки получает дату снятия `2026-05-01`.

Если показания в PDF снижаются относительно предыдущей строки, импорт создает отдельный сегмент счетчика. Если в PDF есть явная строка `Новый счетчик`, последний сегмент получает дату установки и серийный номер из PDF, предыдущие сегменты остаются историческими. Если активный счетчик участка уже есть, но его серийный номер пустой, import apply дозаполняет номер из PDF и пишет изменение в `audit_logs`.

Если часть строк пропущена из-за конфликта или неоднозначности, файл остается в статусе `parsed`, а оператор видит warning. Статус `applied` ставится только когда операция прошла без пропущенных доменных действий.

Все созданные доменные сущности и итог применения пишутся в `audit_logs` с источником `import`.

## Применение Пачки

Карточка пачки содержит кнопку `Применить распознанные`, если в пачке есть файлы в статусе `parsed`.

Операция проходит только по распознанным файлам. Каждый файл применяется отдельно через тот же applier, что и кнопка в карточке файла:

- полностью примененный файл получает статус `applied`;
- файл с пропущенными операциями остается в статусе `parsed`, чтобы оператор мог открыть его и разобрать предупреждения;
- при технической ошибке массовое применение останавливается, а уже примененные файлы остаются примененными.

Это не механизм фоновой очереди. Для исторической загрузки сотен PDF оптимальный путь: staging через CLI или небольшими web-пачками, затем подтверждение применения из карточки пачки или через CLI apply.

## Зависимость От PDF

Разбор PDF делается через `pdftotext -layout`, поэтому в Docker image добавлен пакет `poppler-utils`.

Если контейнер уже был собран до добавления пакета, его нужно пересобрать. До пересборки команда в контейнере сможет работать только с `--from-text`.

## Текущий Формат Результата

Dry-run JSON содержит:

- `source` - формат и режим разбора;
- `account.number` - номер участка в формате вроде `9-123`;
- `subscriber.full_name` - ФИО из квитанции;
- `electricity_meter` - дата установки нового счетчика, серийный номер и начальное показание, если они есть в PDF;
- `rows` - помесячные строки акта сверки;
- `totals` - общие начисления, оплаты и баланс из итоговой строки;
- `payment_requisites` - реквизиты и назначение платежа из формы ПД-4;
- `warnings` - признаки неполного разбора.

## Будущий Import Flow

Следующий безопасный путь:

1. Загрузка пачки PDF в отдельный custom-раздел, доступный только для нужного хозяйства.
2. Создание import batch и сохранение результата dry-run в staging, без создания доменных сущностей.
3. Экран предпросмотра: что будет создано или сопоставлено с уже существующими участками, абонентами, счетчиками, платежами и начислениями. Статус: реализован как read-only preview в карточке файла.
4. Ручное подтверждение оператором. Статус: реализована кнопка `Применить` в карточке файла.
5. Создание доменных сущностей только после подтверждения. Статус: реализовано для участков, абонентов, связей, реквизитов, справочников однотарифного импорта, тарифных периодов/ставок, правил соцнорм, счетчиков, показаний, оплат и исторических начислений.
6. Запись в `audit_logs` всех примененных изменений. Статус: реализовано.
7. Массовое применение всех распознанных файлов пачки. Статус: реализовано синхронно поверх staging.
8. CLI-применение staging-пачки. Статус: реализовано командой `app:zm:apply-statement-import-batch`.

Реализованные staging-сущности:

- `zavety_michurina_statement_import_batches` - пачка загрузки;
- `zavety_michurina_statement_import_files` - отдельный PDF, его hash, имя файла, статус разбора и JSON dry-run.

Потенциальные следующие staging-сущности:

- `zm_statement_import_matches` - предложенные сопоставления с участками/абонентами/счетчиками;
- `zm_statement_import_applied_operations` - детализированный журнал примененных операций, если текущих `audit_logs` будет недостаточно для UI отката/разбора.

## Ограничения

- Parser основан на текущем известном PDF-layout и должен проверяться на разных реальных квитанциях.
- Исторические данные из PDF являются snapshot-копией ручного учета, а не первичным банковским источником.
- Автоматически создавать деньги из PDF нельзя без подтверждения оператора.
- Импорт PDF не создает квитанции КомУчета. Он только восстанавливает данные, на основе которых система позже формирует собственные квитанции.
- Итоговая сумма из PDF должна использоваться для сверки с пересчитанным остатком к оплате, а не как самостоятельное финансовое значение без проверки.
- Оригинальные PDF содержат персональные данные и не должны попадать в Git.
